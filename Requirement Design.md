# 售前知识工具需求规格 V2.0（精简版）

## 1. 产品目标

面向公司售前人员，根据一句话、PDF、Word、Excel、图片及其组合材料，整理客户需求并生成：

1. 可编辑的 Excel 报价单；
2. 面向客户、通俗易懂的 Word 产品文档。

系统优先保证：

- 需求可在网页中确认和修改；
- 重复需求不会重复报价；
- 冲突、缺失信息和来源可见；
- 工时符合约束并尽量命中目标含税报价；
- 输出文件可编辑、可重新下载。

本期按公司内部、单实例、低并发工具设计，不建设企业级平台能力。

## 2. 使用对象与边界

主要用户为公司售前人员。系统不设置审批流程，用户确认报价方案后即可导出正式文件。

本期明确不包含：

- 登录、鉴权、权限、项目归属和安全审查；
- 客户或公司内部审批；
- 多实例、高可用和任务自动恢复；
- PDF 正式输出；
- 自动发送邮件或消息；
- 付款方式、报价有效期、售后期限等商务条款。

实现约束：

- 使用 FastAPI 和 Vue 3 构建前后端单体应用；
- 使用 SQLite，不使用 MySQL 和 Docker Compose；
- 只保留 `project`、`material`、`generation_run`、`artifact` 四张业务表；
- 只保留项目列表和项目工作台两个前端业务路由；
- 上传材料和导出文件保存在本地项目目录；
- 后端只保留接口层和服务层，不设置独立仓储层。

## 3. 整体流程

### 3.1 需求整理

1. 用户新建项目，填写项目参数、目标报价、角色和单价。
2. 用户输入补充说明，并上传一个或多个需求材料。
3. 系统解析材料，合并重复内容，识别冲突、缺失信息和来源。
4. Agent 生成结构化需求草稿。
5. 用户直接在网页中编辑草稿并确认。

网页草稿是后续报价的唯一确认依据。本期不生成供用户修改后重新上传的 Word 草稿。

### 3.2 报价与交付

1. Agent 根据确认需求生成工作包、参与角色建议和相对复杂度权重。
2. 系统根据目标含税报价、角色和单价直接分配工时。
3. 当前配置可以命中目标区间时，只生成一个推荐方案。
4. 当前配置无法命中时，生成最多三个组合调整方案。
5. 用户选择一个方案。
6. 系统生成 Excel 报价单和 Word 产品文档。
7. 系统保存本次生成快照和导出文件。

## 4. 输入要求

### 4.1 支持格式

- 用户直接输入的一句话或补充说明；
- `.pdf`；
- `.docx`；
- `.xlsx`；
- `.png`、`.jpg`、`.jpeg`；
- 上述内容的任意组合。

旧版 `.xls` 不支持，页面提示用户另存为 `.xlsx`。

### 4.2 内容优先级

发生冲突时按以下优先级处理：

1. 用户本次直接输入的补充说明；
2. 用户在网页中确认的结构化需求；
3. PDF、Word、Excel；
4. 图片识别内容。

PDF、Word、Excel 之间优先级相同。同级材料冲突时不得静默选择，应进入“待确认项”。重复需求必须合并。

### 4.3 识别失败

部分材料无法识别时，继续处理其他有效材料，并记录文件名及失败原因。全部材料无法形成有效需求时，任务失败并要求用户补充材料。

## 5. 项目参数

### 5.1 项目类型

- 新项目；
- 旧项目二开。

旧项目二开由 Agent 根据材料判断可复用功能、现状梳理、数据迁移、接口兼容和回归测试需要，不使用固定折扣。

### 5.2 角色与单价

| 角色 | 默认单价 | 是否必选 |
|---|---:|---|
| 产品 | 800 元/人天 | 是 |
| 前端 | 850 元/人天 | 否 |
| 后端 | 850 元/人天 | 否 |
| 测试 | 750 元/人天 | 是 |

用户可以增加其他角色并填写单价。产品和测试不可删除，前端、后端及其他角色可以删除。

### 5.3 报价参数

- 用户输入的目标报价单位为万元；
- 目标报价必须大于 0；
- 目标报价为含 6% 税费的最终总价；
- 合格区间为目标金额的 97%～103%；
- 报价单位默认“重庆酷小贝软件开发有限公司”；
- 报价日期只保留年月日；
- 客户名称可以为空；
- 项目名称为空时由 Agent 根据核心业务生成。

## 6. 结构化需求草稿

需求使用以下层级：

`模块 → 子模块 → 功能 → 子功能 → 功能描述`

每个需求项至少包含：

- 稳定标识；
- 层级和名称；
- 功能描述；
- 来源引用；
- 是否待确认；
- 是否由系统合理补充；
- 参与角色建议；
- 相对复杂度权重。

用户可以在网页中新增、删除、修改和调整需求项。确认后形成不可变需求快照，后续报价和文档导出只读取该快照。

系统可以补充实现业务目标必需的基础功能，但不得主动增加与核心目标无关的扩展功能。存在歧义的内容必须标为待确认。

## 7. 工时与报价规则

### 7.1 职责边界

Agent 负责：

- 生成合理工作包；
- 判断工作包需要哪些角色；
- 给出相对复杂度权重。

后端程序负责：

- 根据目标价分配工时；
- 校验工时与角色约束；
- 计算金额和税费；
- 生成和排序报价方案。

Agent 不直接填写最终金额。系统不生成独立的基准工时和基准报价。

### 7.2 工时约束

- 工时单位为人天；
- 工时以 0.5 人天为步长；
- 单个角色、单个工作包为 0.5～3 人天；
- 不参与某工作包的角色留空；
- 产品和测试项目总工时分别不少于 0.5 人天；
- 超过 3 人天的工作必须拆成多个有实际含义的工作包；
- 不得为了提高报价制造虚假工作包。

### 7.3 金额计算

- 角色费用＝该角色全部工时×角色单价；
- 工时费合计＝全部角色费用之和；
- 税金＝工时费合计×6%；
- 含税总价＝工时费合计＋税金。

金额保留两位小数。同一报价方案中，同一角色只能使用一个统一单价。

### 7.4 推荐方案

系统先在用户当前选择的角色和单价下，根据工作包权重分配工时。如果存在满足全部工时约束且含税总价处于目标价 97%～103% 的分配，只返回一个推荐方案。

### 7.5 组合调整方案

当前配置无法命中目标区间时，系统生成最多三个组合方案。每个方案都可以同时：

- 调整工作包工时；
- 增加、删除或替换具有实际职责的角色；
- 调整角色单价。

每个方案必须显示相对于用户原始配置的角色、单价和工时变化。方案排序依次考虑：

1. 是否命中目标区间；
2. 调整项数量；
3. 调整幅度；
4. 规则风险。

无法形成合理方案时，展示最接近目标的结果，并说明最低工时、最高工时、角色或功能范围等限制，不强行制造工作内容。

## 8. 正式报价单

输出格式为 `.xlsx`。报价单基于确认模板填充，至少包含：

- 报价标题；
- 客户名称；
- 项目名称；
- 报价单位；
- 报价日期；
- 功能和角色工时明细；
- 各角色工时、单价和费用小计；
- 工时费合计；
- 6% 税金；
- 最终含税总价。

工作表名称、列结构、表头、合并单元格、公式、格式和打印区域以参考模板为准。工时、费用、税金和总计使用 Excel 公式，用户修改后可以自动更新。

报价单只包含用户选中的方案。

## 9. 产品文档

输出格式为 `.docx`，使用非技术语言，至少包含：

1. 项目概述；
2. 建设目标；
3. 使用对象；
4. 整体业务流程；
5. 功能介绍；
6. 数据及外部系统说明；
7. 项目范围与不包含内容；
8. 假设与待确认项。

产品文档不包含内部工时、调价过程、报价方案和技术实现细节。

## 10. 历史与删除

- 每次材料分析和报价生成保存一份不可变 JSON 快照；
- 重试创建新记录，不覆盖旧记录；
- 用户可以查看历史结果并重新下载文件；
- 不支持删除单个历史版本；
- 删除项目时删除全部关联记录和项目文件目录。

## 11. 页面范围

系统只保留：

- 项目列表和新建项目；
- 项目工作台。

项目工作台通过步骤条提供：

1. 项目参数与材料；
2. 结构化需求草稿；
3. 工时与报价方案；
4. 文件导出。

历史记录在工作台内以标签页或抽屉展示。

## 12. 任务与失败处理

耗时任务只有：

- 材料分析与结构化草稿生成；
- 工时分配与报价方案生成。

任务状态只有 `pending`、`running`、`succeeded`、`failed`。失败后用户可以手动重试。应用重启导致运行中任务中断时不自动恢复。

Excel 和 Word 导出同步执行。导出失败不得覆盖已有文件或已选方案。

## 13. 验收标准

1. 支持一句话、PDF、Word、Excel、图片及组合输入。
2. 正确执行来源优先级、重复合并和冲突处理。
3. 用户可以在网页中编辑并确认结构化需求。
4. Agent 输出通过固定结构校验。
5. 工时以 0.5 人天递增，单个角色、单个工作包处于 0.5～3 人天。
6. 产品和测试始终保留，且项目总工时分别不少于 0.5 人天。
7. 当前角色和单价可行时，只生成一个命中目标区间的推荐方案。
8. 当前配置不可行时，展示最多三个组合调整方案或最接近目标的不可行结果。
9. 金额和 6% 税费计算正确。
10. Excel 公式、结构、格式和打印区域符合参考模板。
11. Word 产品文档通俗完整，不包含技术实现和内部调价信息。
12. 历史快照可查看，导出文件可重新下载。
13. 正常完整流程和任务失败重试流程通过端到端验证。
