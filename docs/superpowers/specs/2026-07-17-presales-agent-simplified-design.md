# 售前知识工具精简设计

## 1. 目标与范围

建设一个面向公司售前人员的单实例 Web 工具。用户上传一句话、PDF、Word、Excel、图片或其组合材料，在网页中确认结构化需求，系统围绕目标报价分配角色工时，并输出可编辑的 Excel 报价单和面向客户的 Word 产品文档。

本设计以实现简单、单实例、低并发为前提。不实现登录、权限、项目归属、安全审查、审批流程、多实例、高可靠任务队列和任务断点恢复。

## 2. 用户流程

1. 用户新建项目，填写目标报价、角色和单价，并上传需求材料。
2. 系统解析材料，合并重复内容，识别冲突、缺失信息和来源。
3. Agent 生成结构化需求草稿。
4. 用户直接在网页中编辑并确认草稿；不再下载、编辑和重新上传 Word 草稿。
5. 系统根据目标含税总价、确认后的功能结构、角色和单价分配工时。
6. 当前配置能够命中目标价上下 3% 时，只生成一个推荐方案。
7. 当前配置无法命中时，系统生成最多三个组合调整方案。每个方案均可同时调整工时、角色和单价。
8. 用户选择方案后，系统生成 Excel 报价单和 Word 产品文档。
9. 系统保存本次生成快照，供查看和重新下载。

## 3. 系统架构

- FastAPI 提供 JSON API、文件上传、文件下载，并托管 Vue 构建后的静态页面。
- Vue 3 负责项目录入、结构化草稿编辑、方案选择和文件下载。
- SQLite 保存业务数据。
- 本地 `storage/{project_id}` 目录保存上传材料和导出文件。
- SQLAlchemy 负责数据库访问，Alembic 负责数据库迁移。
- 后端只保留接口层和服务层，不设置独立仓储层。
- 外部 Agent 调用通过一个窄接口隔离，自动化测试使用假实现。
- 不使用 MySQL、PyMySQL、Docker Compose、Redis、Celery、Pinia 和 Axios。

耗时任务只有两类：

1. 材料分析与结构化草稿生成；
2. 工时分配与报价方案生成。

Excel 和 Word 导出同步执行。首版使用应用进程内后台任务；进程重启时正在运行的任务可能中断，用户通过手动重试创建新任务。

## 4. 数据模型

系统只保留四张核心表。

### 4.1 `project`

保存项目名称、客户名称、项目类型、目标含税报价、报价单位、报价日期、角色与单价 JSON、当前阶段、当前选中的生成记录与方案标识，以及创建时间。

### 4.2 `material`

保存项目 ID、原始文件名、存储路径、文件类型、解析状态、来源优先级、解析摘要和创建时间。

### 4.3 `generation_run`

保存项目 ID、任务类型、任务状态、结构化草稿 JSON、确认需求 JSON、工时与报价方案 JSON、错误信息、Agent 模型信息和创建时间。

每次成功生成保存一份不可变快照。用户选择方案时只更新 `project` 指向的生成记录和方案标识，不修改快照；重试创建新记录，不覆盖旧记录。

### 4.4 `artifact`

保存项目 ID、生成记录 ID、文件类型、版本号、文件路径和生成时间。文件类型仅包括 Excel 报价单和 Word 产品文档。

金额统一按“分”存储为整数。工时按“半天单位”存储为整数，例如 `3` 表示 `1.5` 人天。税率固定为 6%，最终展示时转换为两位小数。

## 5. 需求草稿结构

需求按照以下层级组织：

`模块 → 子模块 → 功能 → 子功能 → 功能描述`

每个需求项至少包含：

- 稳定标识；
- 层级与名称；
- 功能描述；
- 来源引用；
- 是否待确认；
- 是否由系统合理补充；
- 参与角色建议；
- 相对复杂度权重。

用户在网页中编辑结构化字段。确认后形成不可变需求快照，后续报价和文档导出只能读取该快照。

## 6. 报价分配规则

### 6.1 职责边界

Agent 负责生成工作包、判断参与角色和给出相对复杂度权重。后端程序负责工时分配、金额计算、约束校验和方案排序。Agent 不直接决定最终金额。

### 6.2 当前配置分配

1. 将目标含税价换算为未税预算。
2. 根据工作包复杂度权重和参与角色分配工时。
3. 工时以 0.5 人天为步长。
4. 单个角色、单个工作包为 0.5 至 3 人天；不参与时留空。
5. 产品和测试角色始终保留，项目总工时分别不少于 0.5 人天。
6. 不得增加与确认需求无关的工作包。
7. 后端计算工时费、6% 税金和含税总价。

当前角色和单价下存在命中目标价上下 3% 的合理分配时，只返回一个推荐方案。

### 6.3 组合调整方案

当前角色和单价无法命中时，生成最多三个组合方案。方案可以同时：

- 调整工作包工时；
- 增加、删除或替换具有实际职责的角色；
- 调整角色单价。

每个方案必须展示相对于用户原始配置的角色、单价和工时变化。方案排序依次考虑：是否命中目标区间、调整项数量、调整幅度和规则风险。

如果仍无法形成合理方案，返回最接近目标的结果，并说明受最低工时、最高工时、角色约束或功能范围限制而不可行。系统不得为了命中价格制造虚假功能或无职责角色。

系统不生成和保存独立的“真实基准工时”。

## 7. 页面设计

系统只保留两个前端路由：

- `/projects`：项目列表和新建项目；
- `/projects/:id`：项目工作台。

项目工作台使用步骤条组织四个区域：

1. 项目参数与材料；
2. 结构化需求草稿；
3. 工时与报价方案；
4. 文件导出。

历史记录作为项目工作台中的标签页或抽屉，不设置独立页面。历史记录只支持查看和重新下载，不支持删除单个版本。删除项目时同时删除全部数据库记录和项目文件目录。

## 8. 状态与错误处理

生成任务状态只有：`pending`、`running`、`succeeded`、`failed`。

项目阶段只有：`input`、`draft_ready`、`quote_ready`、`completed`。

- 材料部分解析失败时继续处理其他有效材料，并把失败文件列入待确认项。
- 全部材料无法形成有效需求时，任务失败并要求用户补充材料。
- Agent 输出必须先通过固定 Pydantic 模型校验；校验失败时任务失败。
- 任务失败时保存可读错误信息，用户重试时创建新的 `generation_run`。
- 导出失败不得覆盖已有文件或已选方案，用户可以再次执行导出。
- 应用重启后遗留的 `running` 任务不自动恢复，由用户手动重试。

## 9. 测试与验收

自动化测试重点覆盖：

- 目标价分配、0.5 人天步长、角色约束、税费和金额计算；
- 当前配置可行、当前配置不可行、组合方案和完全不可行场景；
- PDF、Word、Excel、图片解析以及来源优先级；
- Agent 结构化输出校验；
- Excel 公式、模板结构和打印区域；
- Word 标题结构、功能覆盖和待确认标记；
- 正常完整流程和后台任务失败重试流程。

不要求每个 Vue 组件和每个实施切片都有独立测试。语义类验收使用固定样例和人工检查，重点确认需求未明显遗漏、重复功能已合并、来源可追溯、文档语言面向非技术客户。

## 10. 明确不包含

- Word 草稿下载后修改并重新上传；
- 独立基准工时和基准报价；
- 固定的工时、人员、单价三类独立报价策略；
- 单个历史版本删除；
- 登录、鉴权、权限、项目归属和数据安全审查；
- 多实例部署、可靠任务队列、自动任务恢复；
- PDF 正式输出、审批、支付和消息通知。
